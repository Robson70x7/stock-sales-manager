# Funcionalidades do Mobile a Implementar no Desktop

> Documento de referência com as funcionalidades implementadas na versão mobile
> (Expo/React Native) que ainda não existem no desktop (Electron/React).

## 1. `entryPaymentType` — Tipo de Pagamento da Entrada

### Descrição
Permite que o valor de **entrada** de uma venda parcelada tenha um tipo de
pagamento **diferente** do restante das parcelas. Exemplo prático:

- Cliente compra um produto de R\$ 1.200,00
- Dá R\$ 200,00 de **entrada no PIX**
- Parcela o saldo restante (R\$ 1.000,00) em **5x no cartão de crédito**

### Onde existe no mobile
- **Schema**: coluna `entryPaymentType TEXT` na tabela `sales` (migration
  `005_entry_payment_type.sql`)
- **Types**: campo `entryPaymentType?: PaymentType` no tipo `Sale` e nos tipos
  de resposta do banco (`DbSale`)
- **UI (criação)**: seletor de chips (PIX, Dinheiro, Cartão, etc.) exibido
  abaixo do campo "Valor da entrada" quando `entryAmount > 0`
- **UI (exibição)**: nas listas e detalhes da venda, a entrada aparece como
  `Entrada · PIX` (ou o tipo correspondente)
- **UI (edição)**: exibido como campo **read-only** (não permite alterar após
  criada)

### O que implementar no desktop
1. Adicionar coluna `entryPaymentType TEXT` na tabela `sales` via migration
2. Adicionar campo no tipo `Sale` no frontend
3. No formulário de venda (`SaleFormPage`), quando `entryAmount > 0`, exibir
   seletor de tipo de pagamento para a entrada (mesmos tipos: PIX, Dinheiro,
   Débito, Crédito, Transferência)
4. Exibir o tipo de pagamento da entrada no **card da parcela de entrada** na
   tela de detalhes
5. Marcar como apenas leitura na edição

### Regras de negócio
- `entryPaymentType` só faz sentido quando `entryAmount > 0` E
  `installmentsCount > 1`
- O tipo de pagamento da entrada é independente do `paymentType` da venda
- Os chips disponíveis são os mesmos que os tipos de pagamento regulares, com
  exceção de "Parcelado" (não faz sentido para entrada)

## 2. `entryAmount` — Entrada com Parcelamento Dedicado

### Descrição
Diferente da versão desktop atual (que não suporta entrada), o mobile permite
que uma venda tenha um **valor de entrada** pago à vista, e o **restante**
dividido em parcelas. A entrada vira uma parcela especial com `number=0`, e as
parcelas regulares vão de 1 a N.

### Detalhamento da lógica de parcelas

```
Valor total:  R$ 1.200,00
Entrada:      R$ 200,00  →  number=0, totalInstallments=5
Parcela 1/5:  R$ 200,00  →  number=1, totalInstallments=5
Parcela 2/5:  R$ 200,00  →  number=2, totalInstallments=5
...
Parcela 5/5:  R$ 200,00  →  number=5, totalInstallments=5
```

- `totalInstallments` é o **número de parcelas regulares** (não inclui a
  entrada)
- Cada parcela regular = `(totalAmount - entryAmount) / installmentsCount`
- A entrada tem `number=0` e vence na mesma data da primeira parcela
  (`firstInstallmentDate`)

### Onde existe no mobile
- **Schema**: coluna `entryAmount REAL DEFAULT 0 NOT NULL` na tabela `sales`
- **Cálculo**: função `generateInstallmentsWithEntry()` em `lib/utils.ts`
- **Exibição nas listas**: parcela com `inst.number === 0` é exibida como
  "Entrada" (com label diferenciada)
- **Exibição nos detalhes**: label "Entrada" com data de vencimento e valor

### O que implementar no desktop
1. Adicionar coluna `entryAmount REAL DEFAULT 0` na tabela `sales`
2. Reformular a geração de parcelas:
   - Função atual (`generateInstallments`) divide `totalAmount /
     installmentsCount`
   - Nova função (`generateInstallmentsWithEntry`) deve subtrair `entryAmount`
     antes de dividir
3. No formulário de venda:
   - Campo "Valor da entrada" (currency input)
   - Quando `entryAmount > 0`, ajustar o cálculo de exibição de parcelas
4. Na tela de detalhes:
   - Mostrar parcela de entrada separadamente (com label visual "Entrada")
   - Indicar visualmente que é uma parcela diferente
5. Na tela de edição:
   - Exibir entrada e parcelas corretamente
   - `entryAmount` não deve ser editável após criação (ou permitir com
     recálculo)

### Regras de negócio
- `entryAmount` não pode ser maior que `totalAmount`
- `entryAmount` deve ser >= 0
- Se `entryAmount > 0`, `installmentsCount` mínimo é 1
- A entrada é considerada "paga" no ato da criação (não precisa de toggle de
  pagamento)

## 3. Histórico de Movimentações de Estoque (`stock_movements`)

### Descrição
Sistema **transacional** de estoque: toda alteração no estoque de um produto
gera um **registro imutável** na tabela `stock_movements`, permitindo auditoria
completa de:

- **Entrada** (compra, ajuste manual para mais)
- **Saída** (venda, ajuste manual para menos)
- Data/hora, quantidade, saldo resultante e observação

O desktop atual apenas altera o campo `stock` do produto sem manter histórico.

### Schema da tabela (mobile)

```sql
CREATE TABLE stock_movements (
  id TEXT PRIMARY KEY,
  productId TEXT NOT NULL REFERENCES products(id),
  type TEXT NOT NULL CHECK(type IN ('in', 'out', 'sale', 'adjustment')),
  quantity INTEGER NOT NULL,
  balanceAfter INTEGER NOT NULL,
  referenceId TEXT,
  notes TEXT,
  createdAt TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### Onde existe no mobile
- **DB**: tabela `stock_movements` com funções `addStockMovement()` e
  `getStockMovementsByProduct()`
- **UI**: tela de detalhes do produto (`app/products/[id].tsx`) com seção de
  "Histórico de Movimentações"
  - Timeline vertical com dots coloridos:
    - Verde = entrada (`in`)
    - Vermelho = saída (`out`)
    - Azul = venda (`sale`)
    - Amarelo = ajuste (`adjustment`)
  - Cada item mostra: data formatada, tipo (label em português), quantidade
    com sinal (+/-), saldo resultante e observações
  - Ordenado do mais recente para o mais antigo

### O que implementar no desktop
1. Criar migration `003_stock_movements.sql` com a tabela
2. Integrar movimentações nos pontos existentes:
   - **Ao criar venda**: registrar saída para cada produto vendido (type =
     'sale', quantity negativo, referenceId = saleId)
   - **Ao cancelar venda**: registrar entrada de volta (type = 'sale',
     quantity positivo, referenceId = saleId)
   - **Ao excluir venda**: se `returnStock=true`, registrar entrada
   - **Ao editar produto (estoque manual)**: se o estoque mudou, registrar
     ajuste (type = 'adjustment', com notes explicando)
3. Na tela de detalhes do produto:
   - Adicionar seção "Histórico de Movimentações" abaixo dos stats
   - Usar timeline visual (pode ser com shadcn/ui components)
   - Mostrar badges coloridos por tipo
   - Formatar data relativa (ex: "há 2 dias")
4. Funções no `DatabaseService.ts`:
   - `addStockMovement(productId, type, quantity, balanceAfter, referenceId?,
     notes?)`
   - `getStockMovementsByProduct(productId)`

### Regras de negócio
- `balanceAfter` é o saldo **após** a movimentação (calculado no backend)
- `sale` usa `quantity` negativo para saída, positivo para retorno
- `adjustment` pode ser positivo ou negativo
- Ao excluir uma movimentação de venda (raro), manter consistência

## 4. Filtro por Tipo de Pagamento + Ordenação no Resumo (Dashboard)

### Descrição
A tela de Resumo (Dashboard) do mobile ganhou modal de filtros avançados e
modal de ordenação que não existem no desktop.

### Funcionalidades do mobile

#### Modal de Filtros
- Grid de **chips de tipo de pagamento** (PIX, Dinheiro, Crédito, Débito,
  Transferência) — multi-select
- Botões "Limpar" e "Aplicar"
- Chips ativos exibidos abaixo da barra de busca, com botão "x" para remover
  individualmente

#### Modal de Ordenação
- Opções radio-style:
  - **Data (padrão)**: mais recentes primeiro
  - **Maior valor**: do maior para o menor
  - **Menor valor**: do menor para o maior
- Botão no header ao lado do botão de filtro

### Onde existe no mobile
- `app/(tabs)/index.tsx`:
  - Estado `filters` (paymentTypes, sortBy)
  - `filteredItems` com useMemo aplicando filtro de payment type + sort
  - Modal `FilterResumoModal` e `SortResumoModal`

### O que implementar no desktop
1. No `Dashboard.tsx`:
   - Adicionar **dois botões no header** (ao lado da busca): "Filtrar" (ícone
     de filtro) e "Ordenar" (ícone de setas)
   - **Modal de Filtro** (shadcn Dialog):
     - Seção "Tipo de Pagamento" com chips multi-select (PIX, Dinheiro,
       Crédito, Débito, Transferência)
     - Botões "Limpar" e "Aplicar"
   - **Modal de Ordenação** (shadcn Dialog):
     - Radio group: "Data", "Maior valor", "Menor valor"
     - Botão "Aplicar"
   - **Chips de filtro ativos** entre a busca e a lista:
     - Cada chip com label + botão "x" para remover
     - Quando houver filtros ativos, exibir "Limpar todos" no final
2. Lógica de filtragem:
   - Combinar filtro de texto existente + filtro de payment type + sort
   - Payment type filtra itens cujo `installmentInfo?.paymentType` está no
     array selecionado
   - Sort: ordenar por data (padrão), `amount` descending ou `amount` ascending

## 5. Opção "Venda já paga" no Cadastro

### Descrição
Checkbox no formulário de criação de venda que permite marcar a venda como
**paga no ato da criação**, para pagamentos à vista que não são cartão de
crédito. No desktop, o status só pode ser alterado manualmente depois na tela
de edição.

### Comportamento no mobile
- Exibido apenas quando **não há parcelamento** (`installmentsCount <= 1` ou
  `paymentType` não é "credit_card")
- Checkbox com label "Venda já paga"
- Quando marcado:
  - A parcela única é criada com `status: 'paid'`
  - O `paidDate` da parcela é preenchido com a data atual
  - O status da venda (`sale.status`) é salvo como `paid`
- Quando desmarcado (padrão):
  - A parcela única é criada com `status: 'pending'`
  - O status da venda é salvo como `pending`

### Onde existe no mobile
- `app/sales/new.tsx`: state `isPaid`, checkbox renderizado condicionalmente
- `handleSave`: passa `isPaid` para o status da venda e da parcela

### O que implementar no desktop
1. No `SaleFormPage.tsx`:
   - Adicionar estado `isPaid` (boolean, default false)
   - Renderizar checkbox com label "Venda já paga"
   - Condições de exibição:
     - `paymentType !== 'credit_card'`
     - `installmentsCount <= 1`
     - Ou: sempre que houver apenas 1 parcela
   - Ao salvar:
     - Se `isPaid`: `installments[0].status = 'paid'`,
       `installments[0].paidDate = new Date().toISOString()`,
       `saleStatus = 'paid'`
     - Se não: `installments[0].status = 'pending'`, `saleStatus = 'pending'`

### Regras de negócio
- Se a venda tiver entrada (`entryAmount > 0`), a entrada é SEMPRE considerada
  paga, mas o checkbox controla apenas as parcelas regulares
- Para cartão de crédito, mesmo com 1 parcela, não exibir o checkbox (pagamento
  futuro)
- Se o usuário marcar "já pago", o `sale.status` no banco deve ser `paid`
  (consistente com filtros e relatórios)

## 6. Busca com TextInput nos Modais de Seleção

### Descrição
Os modais de seleção de **cliente** e **produto** no formulário de venda
possuem um campo de busca textual que filtra a lista em tempo real. O desktop
já possui isso (searchable picker modal), mas vale listar para manter a
paridade.

### Status no desktop
- ✅ Já implementado nos modais de `ClientPicker` e `ProductPicker`
- Caso precise de melhorias: adicionar placeholder "Buscar por nome..." e
  debounce de 300ms

## 7. Sistema P2P Sync (mDNS + WebSocket) — Mobile Only

### Descrição
O mobile possui um sistema de **sincronização peer-to-peer local** via mDNS
(descoberta de dispositivos na rede) + WebSocket (transferência de dados). Isso
permite que dois dispositivos na mesma rede local sincronizem dados sem
servidor central.

### Status no desktop
- ❌ Não implementado
- O desktop é totalmente offline/local
- **Requer avaliação**: faz sentido implementar sync no desktop? O mobile sync
  foi pensado para dispositivos móveis que ficam o dia todo na loja. O desktop
  normalmente é o "servidor" principal.

### Sugestão
- Se implementar no desktop, torná-lo o **hub central** de sync (recebe dados
  dos mobiles e consolida)
- Usar o mesmo protocolo WebSocket + mesmo formato de mensagens do mobile
- Os tipos e constantes compartilhados estão em `shared/sync/`

## 8. Auth / Login — Mobile Only

### Descrição

### Status no desktop

- O desktop é single-user e local
- **Requer avaliação**: apenas se houver plano de multiusuário ou servidor
  remoto
